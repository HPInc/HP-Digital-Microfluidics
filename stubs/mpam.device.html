class mpam.device.BinaryComponent(board: Board, *, state: State[OnOff], live: bool = True)ÔÉÅ

Bases: BoardComponent, OpScheduler[BC]

A subclass of BoardComponent with an OnOff state.

This class is parameterized by the actual subclass (BC) so that it can inherit from OpScheduler[BC]. For example:

class Magnet(BinaryComponent['Magnet']): ...

This means that Magnet can schedule() operations of type Operation[Magnet, V].

Each BinaryComponent objects is associated with a State[OnOff] object (state), which it delegates to to implement current_state and on_state_change().

A BinaryComponent may be broken, in which case attempting to schedule one of its ModifyState operations will result in PadBrokenError being raised.

It can also be flagged as not being live. The distinction between broken and live is somewhat confused, but the initial intention was that live was used to identify pads on a board that couldn‚Äôt be used by design and broken was for pads that were discovered to be non-functional.

class mpam.device.GateStatus(value)ÔÉÅ

Bases: Enum

The status of WellGates during a WellMotion transition. This is used to determine whether it is safe for another Well in the same DispenseGroup to piggyback on the motion when going to the same target WellState. This takes advantage of the fact that shared WellPads on all Wells in the DispenseGroup turn on and off together, so except for the gate, the Well has been following along. If it is safe to do so, this can result in efficiencies by having multiple Wells dispense or absorb at the same time, even if the requests are staggered.

If the status is NOT_YET, the gates have not yet turned on, so it is known to be safe. If it is JUST_ON, it is safe, but the new Well‚Äôs gate needs to be scheduled to turn on. If it is UNSAFE, the Well cannot piggyback and must wait until the current motion is done.

class mpam.device.Pad(loc: XYCoord, board: Board, state: State[OnOff], *, exists: bool = True)ÔÉÅ

Bases: BinaryComponent[Pad], DropLoc, LocatedPad

A pad, typically in the main array of its associated Board (i.e., not in a Well).

  • As a BinaryComponent, it has a board and a current_state, it can act as both a CommunicationScheduler and a OpScheduler[Pad], and it supports on_state_change(). It also has an unchangeable version of live (exists) that can be used to indicate that not only is the Pad not currently live, but it never will be.

    To initialize this asspect of the Pad, when it is created, it must be associated with a State[OnOff] object that knows how to effect any changes on a physical device (via its realize_state() method). If nothing needs to be done, a DummyState may be used.

  • As a DropLoc, it had an optional drop and blob, and it supports on_drop_change(). It is empty whenever drop is None.

  • As a LocatedPad, it has a location, a row, and a column.

In addition, a Pad may be associated with a Magnet (magnet), a Well (well, i.e., as the Well‚Äôs exit_pad), a Heater (heater), an ExtractionPoint (extraction_point), and a dried Liquid (dried_liquid).

Whenever the current_state changes, board‚Äôs journal_state_change() is called with the (necessarily different) old and new values to prep for drop motion inference.

Neighbors:

A Pad can enumerate neighboring Pads in several ways:

  • neighbor() takes a Dir and returns the neighboring pad in that direction, if one exists.

  • all_neighbors is the (up to 8) neighboring pads in all directions.

  • neighbors is the (up to 4) neighboring pads in the cardinal directions.

  • corner_neighbors is the (up to 4) neighboring pads in the non-cardinal directions.

  • compute_neighbors_for_blob() (the implementation of neighbors_for_blob) is neighbors plus the gate of well (if it isn‚Äôt None).

  • between_pads is a mapping from Pads two steps away in any cardinal direction to the Pad one step away (i.e., the Pad between that one and this one).

Traffic Control and Reservation:

The Pad provides a rudimentary form of traffic control for Drop motion. The basic notion is that

  • A thread participating in this traffic control will only move a Drop onto a Pad if it has successfully called reserve() on it. This will only succeed if reserved is False, and it will have the side-effect of setting reserved to True.

  • Before calling reserve(), the thread will first determine that the Pad is safe (or safe_except() for the Pad or Well that is the source of the motion). A Pad is safe if it is empty and none of its neighboring Pads are reserved or have a Drop.

  • After the Drop has moved to the Pad, the thread should set :reserved` to False.

Warning

This protocol is not thread-safe. It probably should be made to be. The original expectation was that it would only be run from within the ClockThread

Example

The following code might be used to move a Drop from a source Pad to a target Pad. The Board.before_tick() method takes a Callable that returns None when it is done and a number of Ticks to wait until trying again when it is not.

def do_it() -> Iterator[Optional[Ticks]]:
    one_tick = 1*ticks

    while not target.safe_except(source):
        yield one_tick
    while not target.reserve():
        yield one_tick

    with system.batched():
        target.schedule(Pad.TurnOn)
        source.schedule(Pad.TurnOff)

        def unreserve() -> None:
            target.reserved = False

        board.after_tick(unreserve)

    return None


iterator = do_it()
board.before_tick(lambda: next(iterator))

A well.

  • As a BoardComponent, it has a board, and it can act as a CommunicationScheduler.

  • As a PipettingTarget, it participates in the pipetting protocol by defining prepare_for_add(), prepare_for_remove(), pipettor_added(), and pipettor_removed().

Callbacks registered by on_liquid_change() are fired both when contents is set and on calls to transfer_in() and transfer_out(). Note that on the transfer functions, the same value will be passed as both the old and new parameters. To monitor reagent or volume changes, add callbacks to the Liquid itself using its on_reagent_change() or on_volume_change().

When a Drop.DispenseFrom Operation is scheduled, before making the request, it loops calling reserve_gate until it succeeds. When the dispense is finished, it sets gate_reserved to False. This serializes multiple requests to dispense from the same Well.

Most of the actual interaction with a Well involves querying or modifying its contents, which is either a Liquid or None, which is the initial state before any Liquid is put in the well.

  • The volume and reagent of the contents can be obtained directly (but not modified). If contents is None, these are zero and unknown_reagent, respectively.

  • A Well has a capacity and a remaining_capacity (defined as capacity - volume).

  • A Well has a dispensed_volume (the nominal size of a dispensed Drop). Its drop_availability is the number of Drops of that size it can be dispensed. It is considered to be empty if there is less than one Drop available.

  • If is_voidable (a constant property) is True, when the volume becomes exactly zero, it is considered to have been voided, and it is safe for the Well to be reused for a different Reagent.

  • A Well is available if either its contents is None or it is_voidable and its volume is zero and its contents is not inexact.

  • To test whether a Well is compatible with a given Reagent, use can_provide() and can_accept(). Both of these return True if the proffered Reagent matches the Well‚Äôs reagent, if the Well is available or if either Reagent is the unknown_reagent. In addition, can_accept() returns True if the Well‚Äôs reagent is the waste_reagent, and can_provide() returns True if the queried Reagent is the waste_reagent.

  • To assert the contents of a Well, you can set contents directly (which simply adopts the proffered optional Liquid) or call contains(), specifying the Liquid or Reagent. While the former is the only way to reset contents to None, the latter is safer in other cases, as it both creates a new Liquid and clips the volume at capacity. It also takes an optional ErrorHandler that can be used to specify an action to take when the volume of a Liquid argument is greater than capacity. (The default is simply to print a message.)

  • To assert that Liquid has been added to or removed from a Well, call transfer_in() or transfer_out(). Note that these functions are not typically called directly. Rather, they are called as part of the implementation of drop motion inference and pipetting actions.

  • To request that Liquid be added to or removed from a Well, call add() or remove(). The default implementation turns around and asks its associated pipettor to do the work. Note that if pipettor is None, these operations will fail.

  • Calling refill() calls add() to add Liquid until volume reaches the fill_to level. This level is the first of

    • a Volume set as the value of fill_to,

    • the result of calling a function passed to compute_fill_to(),

    • if required is not None, the minimum of it and capacity. required is an optional Volume that can be set to an overall amount of Liquid that will be needed from the Well. It is updated by transfer_out().

    • the Well‚Äôs capacity.

  • Similarly, calling empty_well() calls remove() to remove Liquid until volume reaches the empty_to level. This level is the first of

    • a Volume set as the value of empty_to,

    • the result of calling a function passed to compute_empty_to(), or

    • zero

iterator() → Iterator[bool]ÔÉÅ

Take one or more Wells to the target WellState. Yields True` whenever it should be asked to continue before the next clock tick.  Yields ``False if either it has finished or it has been adopted by another in-progress WellMotion. If it has finished, before yielding, it schedules calling post_futures() after the current tick.

The basic procedure is:

  1. Wait (yielding True) for any guard to signal that it is safe to proceed.

  2. If the group has a motion, see if it can adopt us. If we can, yield False.

  3. If it can‚Äôt, try again next tick.

  4. If there is no motion in the group, set ourselves to be it.

  5. Grab a WellOpStepSeq using the group‚Äôs transition_func.

  6. Walk through each WellOpStep in the sequence, setting pads to their stated values and updating gate_status.

    • At any point here, another WellMotion may call try_adopt() on us to get us to adopt them.

  7. When we are done, schedule calling post_futures() for after the next clock tick and yield False to signal that we‚Äôre done.
