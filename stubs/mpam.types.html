class mpam.types.AsyncFunctionSerializer(*, thread_name: Optional[str] = None, daemon_thread: bool = False, before_task: Optional[Callable[[], Any]] = None, after_task: Optional[Callable[[], Any]] = None, on_empty_queue: Optional[Callable[[], Any]] = None, on_nonempty_queue: Optional[Callable[[], Any]] = None)

Bases: object

Calls functions in a background thread. Functions are added by means of enqueue() and are called sequentially.

The AsyncFunctionSerializer can specify actions to be performed around the enqueued functions:

  • before_task: called before each function

  • after_task: called after each function

  • on_nonempty_queue: called when the queue goes from being empty to being nonempty

  • on_empty_queue: called when the queue goes from being nonempty to being empty

Note that it is legal for these actions to call enqueue().

The background thread is only created when the queue becomes non-empty, and it goes away when the queue becomes empty.

By default, the background thread is not a daemon thread, and so the process will not die as long as there are items in the queue. This can be altered by specifying daemon=True when the AsyncFunctionSerializer is initialized.

class mpam.types.ColorAllocator(*, initial_reservations: Optional[Mapping[H, Union[Color, str, tuple[float, float, float], tuple[float, float, float, float]]]] = None, all_colors: Optional[Sequence[str]] = None)

Bases: Generic[H]

An allocator associating Color objects with elments of some Hashable type H.

The basic idea is that you call get_color() to obtain a Color to use to represent an element. For example, if alloc is a ColorAllocator[Reagent] and r is a Reagent, then

c = alloc.get_color(r)

will obtain a Color for r. If called again with the same argument, the same Color will be returned. To reserve a specific Color for an object, use reserve_color():

alloc.reserve_color(waste_reagent, "black")

The color argument to reserve_color() may be a Color or anything usable as the argument to Color.find() (e.g., a string or tuple of floats).

By default, get_color() will allocate Colors from ColorAllocator.default_color_list(), which is an array of 954 common colors that can be found at https://xkcd.com/color/rgb/. To use a different list (or a permutation of this list), specify the all_colors parameter when constructing the ColorAllocator.

Internally, the class uses a WeakKeyDictionary to manage the associations, so it does not hold onto the objects for which Colors have been allocated, and when one gets garbage collected, the associated Color may become available to be associated with a new object. If a color is associated with more than one object (via reserve_color()), it does not become available until all such associations have been dropped.

The colors are allocated in order, skipping any that have been reserved. Only when the list has been exhausted will colors that had been previously allocated for no-longer-extant objects be reused. If all colors are associated with objects, IndexError will be raised.

To remove any Color association for an object, use release_color_for(). This will render the Color (if any) available for future calls to get_color(). Note that release_color() for can safely be called even if its argument was never used in a call to get_color().

Warning

Internally, the mapping from objects to Colors is kept in a WeakKeyDictionary. A consequence of this is that if H defines equality as different from identity, non-identical equal objects will get the same Color, but when the first such object gets garbage collected, the association will be removed from the table and any subsequent ones will not find a match and so will get allocated a new color.

Delayed is an abstract class. Instances are created either by using the complete() class method:

dv = Delayed.complete(val)

to create an instance that has its value from the start or by creating a Postable subclass. In the latter case, the value is asserted by calling the Postable’s post() method:

postable.post(val)

The value may only be specified once (although it need not ever be specified). When a value is posted, any registered callbacks will be invoked and passed the posted value.

Delayed is covariant in its type parameter, so a Delayed[Derived] can be treated as a Delayed[Base]. Conversely, the concrete subclass Postable is contravariant in its type parameter, so a Postable[Base] can be treated as a Postable[Derived].

To register a callback, the when_value() method (or its alias, then_call()) or one of the many methods that delegate to it are used

dv.when_value(do_the_stuff)
dv.then_call(lambda val : print(f"got {val}")
dv.post_to(future)
dv.post_transformed_to(future, lambda v: str(v))
dv.post_val_to(future, val)
dv2 = dv.then_schedule(operation)
dv2 = dv.chain(future)
dv2 = dv.transformed(lambda n: n+1)

Note

If the Delayed object already has a value at the time the callback is added, the callback is invoked immediately. Otherwise, it is remembered and invoked in the thread that calls post() on its Postable subclass instance.

All methods that register callbacks return Delayed objects and can, therefore, be chained to add more callbacks:

dv.when_value(fn1)
  .when_value(fn2)
  .post_val_to(dv2, 7)

Note, however, that some of the methods return different Delayed objects that will receive transformed values.

Delayed objects are thread-safe. A Lock is only constructed if a value has not been asserted at the time the first callback is added. (It would be trivial to avoid the lock if Python had an atomic compare-and-swap operation, but alas.)

To obtain the value of a Delayed object, use one of

f(dv.value)        # blocks until a value is asserted

dv.wait()          # blocks until a value is asserted
f(dv.value)        # now guaranteed not to block

if dv.has_value:
    f(dv.value)    # guaranteed to not block

valid, val = dv.peek()    # returns immediately
if valid:
    f(val)         # val is the value

Note that there is a difference between the Delayed object not having a value and having a value of None.

To block until more than one Delayed object have all received values, use join().

class mpam.types.Liquid(reagent: Reagent, volume: Volume, *, inexact: bool = False)

Bases: object

A quantity of a Reagent. Both the volume and the reagent can change over time, and so differnt Liquids may have the same state, but each will retain its own identity.

Callbacks can be hung off ChangeCallbackLists for both volume and reagent by calling

liq.on_reagent_change(note_reagent_change, key=nrc_key)
liq.on_volume_change(note_volume_change)

To delete a callback, use volume_change_callbacks and reagent_change_callbacks directly, e.g.

liq.reagent_change_callbacks.remove(nrc_key)

In addition to volume, a Liquid also has an indication of whether the volume is inexact. Typically, this will be False, but if it is True, it is not safe to assume that, e.g., incrementally removing volume will necessarily have removed all of it.

As a convenience, a Volume can be added to or subtracted from a Liquid, e.g.

liq -= 2*uL

The resulting Volume is clipped at zero.

Liquids can be mixed together in several ways:

  • mix_with() returns the result of mixing a given Liquid with another. Neither Liquid is modified.

  • mix_in() modifies a given Liquid by setting it to the result of mixing in another. The other Liquid gets the reagent of the mixture, but its volume is set to zero.

  • Liquid.mix_together() returns the result of mixing together several

Liquids (or parts of them). None are modified.

For all of these, the resulting Reagent will be proportional to the volumes. For example, after

liq1 = r1.liquid(1*mL)
liq2 = r2.liquid(2*mL)
r1.mix_in(r2)

r1.volume will be 3*mL and r1.reagent will print as 1 r1 + 2 r2. (r2 will have the same reagent, but its volume will be zero.)

The mixing methods all take an optional result parameter, which is either a Reagent or a string. If it is a Reagent, it will be used as the reagent for the mixture. If it is a string, it will be used as the name for the resulting (computed) mixture unless either the reagents of the two liquids are the same (in which case the mixture has the same reagent) or one of them is the waste_reagent (in which case the mixture will be, as well).

To split the contents of a Liquid into two parts, use split_to(). As this overwrites the content of the other Liquid, it should only be used when the other is known to be empty. It is particularly useful when doing a “merge and split”, as

liq1.mix_in(liq2)
liq1.split_to(liq2)

After this sequence, both Liquids will have the same (mixed) reagent and a volume that is the average of the two original volumes.

A mixture of two Reagents in a given ratio.

The composition of the resulting Reagent will be computed based on the same ratio of the compositions of r1 and r2.
