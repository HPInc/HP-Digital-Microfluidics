mpam.device.Well

class mpam.device.Well(*, board: Board, number: int, group: Union[DispenseGroup, Callable[[WellState, WellState], Iterator[tuple[bool, Sequence[int]]]]], exit_pad: Pad, gate: WellGate, shared_pads: Sequence[WellPad], capacity: Volume, dispensed_volume: Volume, exit_dir: Dir, is_voidable: bool = False, shape: Optional[WellShape] = None)

Bases: OpScheduler[Well], BoardComponent, PipettingTarget

A well.

  • As a BoardComponent, it has a board, and it can act as a CommunicationScheduler.

  • As a PipettingTarget, it participates in the pipetting protocol by defining prepare_for_add(), prepare_for_remove(), pipettor_added(), and pipettor_removed().

Callbacks registered by on_liquid_change() are fired both when contents is set and on calls to transfer_in() and transfer_out(). Note that on the transfer functions, the same value will be passed as both the old and new parameters. To monitor reagent or volume changes, add callbacks to the Liquid itself using its on_reagent_change() or on_volume_change().

When a Drop.DispenseFrom Operation is scheduled, before making the request, it loops calling reserve_gate until it succeeds. When the dispense is finished, it sets gate_reserved to False. This serializes multiple requests to dispense from the same Well.

Most of the actual interaction with a Well involves querying or modifying its contents, which is either a Liquid or None, which is the initial state before any Liquid is put in the well.

  • The volume and reagent of the contents can be obtained directly (but not modified). If contents is None, these are zero and unknown_reagent, respectively.

  • A Well has a capacity and a remaining_capacity (defined as capacity - volume).

  • A Well has a dispensed_volume (the nominal size of a dispensed Drop). Its drop_availability is the number of Drops of that size it can be dispensed. It is considered to be empty if there is less than one Drop available.

  • If is_voidable (a constant property) is True, when the volume becomes exactly zero, it is considered to have been voided, and it is safe for the Well to be reused for a different Reagent.

  • A Well is available if either its contents is None or it is_voidable and its volume is zero and its contents is not inexact.

  • To test whether a Well is compatible with a given Reagent, use can_provide() and can_accept(). Both of these return True if the proffered Reagent matches the Well’s reagent, if the Well is available or if either Reagent is the unknown_reagent. In addition, can_accept() returns True if the Well’s reagent is the waste_reagent, and can_provide() returns True if the queried Reagent is the waste_reagent.

  • To assert the contents of a Well, you can set contents directly (which simply adopts the proffered optional Liquid) or call contains(), specifying the Liquid or Reagent. While the former is the only way to reset contents to None, the latter is safer in other cases, as it both creates a new Liquid and clips the volume at capacity. It also takes an optional ErrorHandler that can be used to specify an action to take when the volume of a Liquid argument is greater than capacity. (The default is simply to print a message.)

  • To assert that Liquid has been added to or removed from a Well, call transfer_in() or transfer_out(). Note that these functions are not typically called directly. Rather, they are called as part of the implementation of drop motion inference and pipetting actions.

  • To request that Liquid be added to or removed from a Well, call add() or remove(). The default implementation turns around and asks its associated pipettor to do the work. Note that if pipettor is None, these operations will fail.

  • Calling refill() calls add() to add Liquid until volume reaches the fill_to level. This level is the first of

    • a Volume set as the value of fill_to,

    • the result of calling a function passed to compute_fill_to(),

    • if required is not None, the minimum of it and capacity. required is an optional Volume that can be set to an overall amount of Liquid that will be needed from the Well. It is updated by transfer_out().

    • the Well’s capacity.

  • Similarly, calling empty_well() calls remove() to remove Liquid until volume reaches the empty_to level. This level is the first of

    • a Volume set as the value of empty_to,

    • the result of calling a function passed to compute_empty_to(), or

    • zero

  • Calling ensure_content() ensures that a Well contains a sufficient amount of Reagent to perform an action. If it doesn’t or if removing the specified Volume would take it below a min_fill level, refill() is called first. min_fill defaults to zero, but can be set to be a Volume or a function that returns one (via :func:`compute_min_fill).

    This is called inside of dispense operations to automatically refill wells when they become empty (or close enough to empty that the hardware does not believe that they can accurately dispense).

  • Similarly, calling ensure_space() ensures that a Well contains a sufficient amount of room to perform an action. If it doesn’t or if adding the specified Volume would take it above a max_fill level, empty_well() is called first. max_fill defaults to capacity, but can be set to be a Volume or a function that returns one (via :func:`compute_max_fill).

    This is called inside of absorbtion operations to automatically emptywells when they become full (or close enough to full that the hardware does not believe that they can accurately absorb).

Request that volume of reagent be added to the Well. Returns a Delayed[Well] that will receive this Well as a value when the action is complete.

Note

The default implementation of add() requires that pipettor not be None. As None is the default, for add() to succeed, it must be set first. If it is not, an assertion will fail.

Note

The actual addition may take place in several steps. If action should be taken on intermediate transfers, use on_liquid_change() to register a callback or override pipettor_added().

Note

If the request cannot be fully satisfied, on_insufficient or on_no_source will be invoked, but he action will still complete.

Warning

If the on_insufficient or on_no_source is invoked, it is likely that this will be in a different thread running the pipettor and raising an exception will cause it to terminate in an unrecoverable way, so RAISE ErrorHandlers should probably not be used.

ensure_content(volume: ~typing.Optional[~quantities.dimensions.Volume] = None, reagent: ~typing.Optional[~mpam.types.Reagent] = None, *, on_reagent_mismatch: ~erk.errors.ErrorHandler = <erk.errors.PRINT_TO object>) → Delayed[Well]

Ensure that volume of reagent can be removed from the Well without taking its volume below its current min_fill level. If this is not the case, refill() is called first.

If it’s not the case that the Well can_provide() reagent, on_reagent_mismatch is invoked.

Reagent matching is checked (and on_reagent_mismatch invoked, if necessary) before the volume is checked, so on some architectures, a mismatch can be fixed by emptying the Well first:

empty_first = FIX_BY(lambda: well.remove(well.volume).wait())
well.ensure_content(reagent=reagent, on_reagent_mismatch=empty_first)

This will empty the Well, setting its volume to zero, and then refill() it.

Note

In order to deal with rounding errors, refill() is not called if the resulting volume would undershoot min_fill by less than 5% of dispensed_volume.

ensure_space(volume: ~quantities.dimensions.Volume, reagent: ~typing.Optional[~mpam.types.Reagent] = None, *, on_reagent_mismatch: ~erk.errors.ErrorHandler = <erk.errors.PRINT_TO object>) → Delayed[Well]

Ensure that volume of reagent can be added to the Well without taking its volume above its current max_fill level. If this is not the case, empty_well() is called first.

If it’s not the case that the Well can_accept() reagent, on_reagent_mismatch is invoked.

Reagent matching is checked (and on_reagent_mismatch invoked, if necessary) before the volume is checked, so on some architectures, a mismatch can be fixed by emptying the Well first:

empty_first = FIX_BY(lambda: well.remove(well.volume).wait())
well.ensure_content(reagent=reagent, on_reagent_mismatch=empty_first)

The downside to this approach is that the Delayed.wait() call will block the thread until the Liquid has been removed. If this is a problem, RAISE should be used instead:

class Mismatch(RuntimeError): ...

try:
    f = well.ensure_space(volume, reagent,
                          on_reagent_mismatch=RAISE(Mismatch))
 except Mismatch:
    f = Delayed[Well]()
    def call_again(_) -> None:
        well.ensure_space(volume, reagent).post_to(f)
    well.remove(well.volume).then_call(call_again)

This catches the mismatch and calls remove(), but schedules a retry of ensure_space() after it is done.

Note

In order to deal with rounding errors, refill() is not called if the resulting volume would undershoot min_fill by less than 5% of dispensed_volume.

remove(volume: ~quantities.dimensions.Volume, *, on_no_sink: ~erk.errors.ErrorHandler = <erk.errors.PRINT_TO object>) → Delayed[Well]

Request that volume of the Well’s reagent be removed from the Well. Returns a Delayed[Well] that will receive this Well as a value when the action is complete.

“Completion” in this sense, is intended to be the point at which the last of the Liquid is removed, not when it reaches its destination.

Note

The default implementation of remove() requires that pipettor not be None. As None is the default, for remove() to succeed, it must be set first. If it is not, an assertion will fail.

Note

The actual removal may take place in several steps. If action should be taken on intermediate transfers, use on_liquid_change() to register a callback or override pipettor_removed().

Note

If the pipettor does not know where to put the Well’s reagent, on_no_sink will be invoked, but the transfer is expected to take place, with the reagent going wherever the pipettor would put the mpam.types.waste_reagent.

Warning

If the on_insufficient or on_no_source is invoked, it is likely that this will be in a different thread running the pipettor and raising an exception will cause it to terminate in an unrecoverable way, so RAISE ErrorHandlers should probably not be used.

Parameters

volume – the Volume of the Well’s reagent to remove

Keyword Arguments

on_no_sink – an ErrorHandler invoked when the Well’s pipettor has no sink for the Well’s reagent

Returns

a Delayed[Well] that will receive this Well when the Liquid has been removed

required: Optional[quantities.dimensions.Volume] = None

The total Volume of Reagent expected to be removed from the Well. May be None.

When refill() is called and required is not None, the computed fill_to level is capped at this value.

When transfer_out() is called, this value is decreased, so it remains an estimate of how much will be needed for the remainder of the run.

transfer_in(liquid: ~mpam.types.Liquid, *, volume: ~typing.Optional[~quantities.dimensions.Volume] = None, on_overflow: ~erk.errors.ErrorHandler = <erk.errors.PRINT_TO object>, on_reagent_mismatch: ~erk.errors.ErrorHandler = <erk.errors.PRINT_TO object>, mix_result: ~typing.Optional[~typing.Union[~mpam.types.Reagent, str]] = None) → None

Adjust the Well’s contents to reflect the addition of liquid. If volume is specified, only that much of liquid is used. If volume is greater than liquid’s Liquid.volume, all of liquid is used.

Note

liquid is adjusted to account for the transfer as well.

If the transfer would result in the capacity of the Well being exceeded, on_overflow is invoked.

If it’s not the case that the Well can_accept() liquid’s reagent and mix_result is None, on_reagent_mismatch is invoked.

After the adjustments, callbacks registered using on_liquid_change() are run, passing in the new contents as both old and new values.

Note

If contents was None, it will first be changed to a volume-zero Liquid with the incoming Reagent. This will also trigger the liquid-change callbacks.

Parameters

liquid – the Liquid that forms the source of the transfer

Keyword Arguments

  • volume – the Volume of liquid transfered or None to indicate all of it.

  • on_overflow – an ErrorHandler for the case in which the transfer would exceed the capacity of the Well

  • on_reagent_mismatch – an ErrorHandler for the case in which liquid’s reagent doesn’t match that of the Well

  • mix_result – the (optional) result of mixing the added Liquid with what’s already there.

transfer_out(volume: ~quantities.dimensions.Volume, *, on_empty: ~erk.errors.ErrorHandler = <erk.errors.PRINT_TO object>) → Liquid

Adjust the Well’s contents to reflect the removal of volume from contents.

If volume is less than volume, on_empty is invoked. After the handler is invoked, volume is clipped to (the possibly altered) volume.

If reagent is not None, volume is subtracted from it. If this would render it zero or negative, it is set to None.

After the adjustments, callbacks registered using on_liquid_change() are run, passing in the new contents as both old and new values.

Note

In order to deal with rounding errors, on_empty is not invoked if volume exceeds volume by less than 5% of dispensed_volume.

Keyword Arguments

on_empty – an ErrorHandler for the case in which volume exceeds the volume of the Well
