class mpam.device.ExtractionPoint(pad: mpam.device.Pad, *, splash_radius: Optional[int] = None)ÔÉÅ

Bases: mpam.types.OpScheduler[ExtractionPoint], mpam.device.BoardComponent, mpam.device.PipettingTarget

A component, (typically a hole in a Board‚Äôs lid) that enables a Pipettor to add Liquid to or remove Liquid from a Pad.

As a BoardComponent, the ExtractionPoint is associated with the same board as its pad.

As a PipettingTarget, ExtractionPoint implements the pipetting protocol:

  • It has an (optional) associated pipettor. By default this is None. A subclass should either provide one or override the implementations of transfer_in() and transfer_out().

  • It can report its removable_liquid. By default, this will be the

contents of the Blob that contains the Drop (if any) at the ExtractionPoint‚Äôs pad.

  • When the Pipettor is in position to add liquid, it calls prepare_for_add() and waits for it to complete. By default, this reserves pad and turns it on.

  • After adding liquid, the Pipettor calls pipettor_added(). By default, this adds the described Liquid, and if this is the last transfer, unreserves pad.

  • When the Pipettor is in position to remove liquid, it calls prepare_for_remove(), which waits until there is a drop on pad.

  • After adding liquid, the Pipettor calls pipettor_removed(), which removes the described Liquid and, if there is no further Liquid in pad‚Äôs blob, turns it off.

To request that Liquid be transfered via an ExtractionPoint, use transfer_in() and transfer_out() (or their wrapper Operations TransferIn and TransferOut). The default implementations of these schedule Pipettor.Supply and Pipettor.Extract on pipettor, failing an assertion if pipettor is None.

When extracting using transfer_out() (or TransferOut), if is_product is True (the default), the system is instructed to treat the extracted Liquid as a product of the overall process and identify a new location (e.g., in an product well plate) to store it. If product_loc is also provided, a ProductLocation (a combination of the Reagent and an implementation-specific description of the location) will be posted to it.

Request that Liquid be extracted through the ExtractionPoint. The method returns a future to which the Drop on pad will be posted after all of the Liquid that will be added has been.

on_no_source will be invoked if a source for reagent cannot be determined. on_insufficient will be invoked if volume of reagent cannot be obtained.

By default, the transfer is implemented by scheduling a Pipettor.Supply Operation on the ExtractionPoint‚Äôs pipettor (which must exist). Subclasses are free to override transfer_in() to remove this restriction.

Request that Liquid be extracted through the ExtractionPoint. The method returns a future to which the Liquid actually transfered will be posted when all of the Liquid that will be removed has been.

If volume is None, the volume of the Blob at pad (which must exist) is used.

If is_product is True, this transfer represents the extraction of a product and a new location (e.g., a new well in a product well plate) will be chosen for it. If product_loc is not None, a ProductLocation will be posted to it describing the Reagent and location.

If a location cannot be determined for the extracted Liquid, on_no_sink will be invoked. If control continues, the Pipettor doing the removal is encouraged to dump the extracted Liquid into a trash receptacle or waste drain.

By default, the transfer is implemented by scheduling a Pipettor.Extract Operation on the ExtractionPoint‚Äôs pipettor (which must exist). Subclasses are free to override transfer_out() to remove this restriction.

Note

The volume of the Liquid posted to the resulting future may be different from volume.

Note

The value will be posted to the resulting future when the last of the liquid is removed, not when it reaches its destination. The value posted to product_loc can happen at any time.
