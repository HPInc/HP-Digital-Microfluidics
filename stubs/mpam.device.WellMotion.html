class mpam.device.WellMotion(well: mpam.device.Well, target: mpam.device.WellState, *, guard: Optional[Iterator[bool]] = None, post_result: bool = True)ÔÉÅ

Bases: object

A transition of (as subset of) the Wells in a DispenseGroup to a target WellState.

The hard work is done by iterator(). This is intended to be used as it is in Well.TransitionTo

motion = WellMotion(well, self.target, post_result=post_result, guard=self.guard)

def before_tick() -> Iterator[Optional[Ticks]]:
    iterator = motion.iterator()
    one_tick = 1*tick
    while next(iterator):
        yield one_tick
    yield None
iterator = before_tick()
board.before_tick(lambda: next(iterator), delta=after)
return motion.initial_future

iterator() → Iterator[bool]ÔÉÅ

Take one or more Wells to the target WellState. Yields True` whenever it should be asked to continue before the next clock tick.  Yields ``False if either it has finished or it has been adopted by another in-progress WellMotion. If it has finished, before yielding, it schedules calling post_futures() after the current tick.

The basic procedure is:

  1. Wait (yielding True) for any guard to signal that it is safe to proceed.

  2. If the group has a motion, see if it can adopt us. If we can, yield False.

  3. If it can‚Äôt, try again next tick.

  4. If there is no motion in the group, set ourselves to be it.

  5. Grab a WellOpStepSeq using the group‚Äôs transition_func.

  6. Walk through each WellOpStep in the sequence, setting pads to their stated values and updating gate_status.

    • At any point here, another WellMotion may call try_adopt() on us to get us to adopt them.

  7. When we are done, schedule calling post_futures() for after the next clock tick and yield False to signal that we‚Äôre done.

try_adopt(other: mpam.device.WellMotion) → boolÔÉÅ

Try to adopt another WellMotion. If successful, other‚Äôs iterator() will yield False, signalling that it is done, but its initial_future will not get a value posted until we call post_futures().

This is called from within iterator(), with group‚Äôs lock locked:

  • after any guard has yielded False

  • after it has been determined that the group‚Äôs motion is this WellMotion.

try_adopt() will fail if

  • the two values of target do not match or

  • our gate_status is UNSAFE or

  • the target is DISPENSED and we are already working on the other‚Äôs Well.
