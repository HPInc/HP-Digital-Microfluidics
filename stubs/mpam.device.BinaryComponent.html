mpam.device.BinaryComponent

class mpam.device.BinaryComponent(board: mpam.device.Board, *, state: Union[mpam.types.State[mpam.types.OnOff], mpam.types.OnOff], live: bool = True, can_toggle: bool = True, on_illegal_toggle: erk.errors.ErrorHandler = PRINT_TO(stdout))

Bases: mpam.device.BoardComponent, mpam.types.OpScheduler[mpam.device.BC]

A subclass of BoardComponent with an OnOff state.

This class is parameterized by the actual subclass (BC) so that it can inherit from OpScheduler[BC]. For example:

class Magnet(BinaryComponent['Magnet']): ...

This means that Magnet can schedule() operations of type Operation[Magnet, V].

Each BinaryComponent objects is associated with a State[OnOff] object (state), which it delegates to to implement current_state and on_state_change.

A BinaryComponent may be broken, in which case attempting to schedule one of its ModifyState operations will result in PadBrokenError being raised.

It can also be flagged as not being live. The distinction between broken and live is somewhat confused, but the initial intention was that live was used to identify pads on a board that couldn’t be used by design and broken was for pads that were discovered to be non-functional.

An Operation that modifies the state of a BinaryComponent.

The ModifyState object is associated with a Modifier function (mod) that’s used to compute the new value based on the old.

The BinaryComponent class (and, therefore, its subclasses) have premade instances for the common cases of TurnOn, TurnOff, and Toggle
